# MakerWorld 本地归档小应用 v4.5 更新说明

## 离线归档页面 (Offline HTML) 架构升级 (v1.0 -> v2.0)
在此版本中，我们对离线单独生成的 `index.html` 进行了全面的底层基础架构更新，以确保本地存档页面和在 Web 服务上查看的 v4.0 页面 (CSR 面板) 拥有**完全一致的视觉和交互体验**。
- **模板内联打包**：摒弃了之前在 `archiver.py` 内通过成百上千行 Python 字符串硬编 HTML 界面的老旧方案。现在的 `archiver.py` 会直接读取前端的 `model.html` 骨架，并将 `variables.css`、`model.css` 以及 `model.js` 的源码全量内嵌入 HTML 输出文件中。真正做到单文件、双击即用的极佳便携性。
- **离线安全的数据注入**：利用 `window.__OFFLINE_META__` 挂载点，将解析好的元数据 (JSON) 原封不动地注入进独立的 HTML 字典。当你在无服务端环境下双击该页面时，原先必须走 `/api/v2/models/...` 进行 Fetch 请求的逻辑会自动判断并跳过，安全直接地渲染出包含了“分盘 Modal 测试、响应图片放大遮罩”的现代 v2.0 页面模型。
- **智能资源定位器**：由前端资源决议器 (`fileUrl`) 分析判断，如果在有服务端环境下，就返回 `/files/{modelDir}/...`；如果存在 `__OFFLINE_META__`，即完全离线，则直接返回纯相对路径 `./...` (如 `./images/...`)，以此规避诸如 Fetch 本地同源拦截和 CORS 跨域痛点。
- **自适应降级控制**：离线模式下，与后台交互紧密关联的“附件上传交互”与“实物成品打印图片上传”块会自动隐形，避免误操作以及毫无意义的 404 失败。

## Windows 平台文件夹空格容错匹配 (Trailing Spaces 404 Fix)
修复了一个由于 Windows 文件夹命名限制引发的严重“404 目录找不到” Bug。
- **Bug 背景**：由于部分模型原采集标题末尾带有空格（或经过 URL `%20` 转码），在采集写入到 Windows 时，Windows 系统的底层 API 会静默吃掉这些尾随空格 (Trailing spaces)。而这会导致浏览器前端使用保留了空格的真实地址 （如 `/v2/files/MW_...%20%20`）在向后端请求时遭遇死胡同。
- **解决策略**：在 `server.py` 的核心解析器 `resolve_model_dir` 内新增 **双级重型 Fallback (回落) 算法**。如果绝对物理路径直接判定文件不存在，系统将：
    1. 首先尝试剔除 URL 传参的首尾空格重试。
    2. 如果依旧失败，暴力遍历整个根目录并读取出其上所有名称字典，忽略所有空格进行对比，直到找到真正的匹配夹，从而在请求侧实现了对尾部空格的绝对免疫。
- **归档优化**：同时在源头根除了该问题——如今采集新建目录时 (`sanitize_filename`)，会直接剥离多余的空格尾巴，降低二次处理压力。
